## What happened in the last session

### 1. The theme changed out from under us
The owner switched the theme in the Chabad One backend. The site now loads
`/css/sites6/purple-theme.css` (it previously loaded `blue-theme.css`).

**Important gotcha:** despite the filename, `purple-theme.css` is **not purple**. It
defines **orange `#f47721`** and **burgundy `#891738`**. The filename is a platform
leftover. Don't trust the name; read the file.

Our custom components were still blue `#0F6CB8` / teal `#2fd3c2`, so they clashed.

### 2. Recolored custom components (in test.css)
Event card: header â†’ orange, title â†’ burgundy, date â†’ orange. Footer phone + social
icons â†’ orange. Focus rings â†’ burgundy. The owner explicitly chose an **outline button
style** (transparent bg, orange border, fills orange on hover) over a solid fill, to
match the platform's own buttons. Preserve that choice unless they say otherwise.

### 3. Built a design token system (the big one)
The owner asked for: one file in the repo controlling all colors, with colors selected by
*what type of element* something is.

Delivered as a `:root` block at the top of `test.css`, in two layers:

- **Raw palette:** `--cc-brand-primary` (orange), `--cc-brand-secondary` (burgundy), plus
  two darker hover variants, and neutrals.
- **Semantic roles:** `--cc-color-header`, `--cc-color-subheading`, `--cc-color-link`,
  `--cc-color-link-accent`, `--cc-color-accent`, `--cc-btn-*`, `--cc-focus-ring`,
  `--cc-footer-accent`.

**Every CSS rule references a ROLE, never a raw hex.** There is zero raw hex outside the
`:root` block, and it should stay that way. Roles are named for what an element *is*, not
what color it is, so the names survive future theme changes.

Two ways to re-theme: edit the raw palette (whole site follows), or repoint a single role
(that category only).

### 4. Native platform overrides
The owner initially wanted the platform's *own* elements to pull colors from our repo
file. **That isn't possible** â€” `purple-theme.css` is served by Chabad One and
regenerated by their backend; we can't edit it. This was explained and accepted.

Instead: our stylesheet loads *after* theirs, so a `NATIVE PLATFORM OVERRIDES` section at
the bottom of `test.css` re-colors their elements using our tokens. Same end result,
different mechanism.

Scope is deliberately the **high-impact set** (~90% of what a visitor sees), not every
element â€” the owner was offered full/high-impact/minimal and this was the recommendation
they approved. Covered: section headers, page/article titles, H2/H3, body links, "more"
links, breadcrumbs, top-bar and nav links, solid buttons, outline buttons, form submit,
mailing-list background, footer accents.

**Known fragility:** these rules target Chabad One's class names. If the platform changes
its markup, a rule can silently stop matching and a stray theme color reappears. That
tradeoff was explained to the owner and accepted.

## THE NEXT PROJECT â€” what the owner wants now

The owner wants a **major design overhaul**. Their words: make the site look "a lot more
updated and more modern and more elegant." They said some of the site's **large/major
features look outdated** and want those specifically addressed.

This is a big, open-ended visual project â€” much larger in scope than anything in the
session above.

**Before writing any code, get specifics.** The brief as stated is not actionable yet.
Things worth pinning down:

- *Which* features feel outdated to them? Likely candidates from the homepage: the hero
  slider, the programs "sneak peek" grid, the latest-photos block, the upcoming-events
  list, the candle-lighting widget, the subscribe/mailing-list section. Get them to name
  the ones that bother them most, ideally in priority order.
- What does "modern and elegant" mean to them concretely? Reference sites they admire are
  far more useful than adjectives.
- How much visual change is acceptable? A restyle within the current structure is very
  different from replacing whole widgets.
- What's non-negotiable? (Their orange/burgundy palette is brand-new â€” confirm whether it
  stays.)

**Constraints that will shape everything:**

- We only control CSS and JS injected after the platform loads. We cannot change the
  platform's HTML structure server-side. Bigger structural changes have to be done by
  JS DOM manipulation after load â€” which is how the existing event card works
  (see the V2 card injection in `live.js` for the established pattern).
- JS-injected elements can flash before styling applies. The blueprint has a rule about
  this: **never scope CSS by `cc-type-*` classes** (JS adds them late). Use native classes
  or `html.cc-aid-XXXXXX` for CSS.
- The token system is now in place â€” **any new component must read from tokens, not
  hardcoded hex.** This is the main architectural rule to preserve.
- Mobile matters. Breakpoint for our custom work is 768px. The platform's default mobile
  spacing is bad (100px padding per row, stacking to ~200px gaps); `test.css` already has
  a mobile spacing fix section.
- The site is a synagogue/community site. Elegant and warm, not corporate or trendy.

**Recommended approach:** pick ONE major feature, redesign it well in `test.css`, get it
previewed and promoted, then move to the next. A single giant overhaul commit would be
very hard for the owner to preview meaningfully and very hard to roll back.
